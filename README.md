# llm-speed-bench
LLMの速度を検証する方法を学ぶ

## 速度に関する指標

### 前提条件
例えばモデルが下記の出力をする場合

> 私はタロウです。

この出力をトークンごとに分けると、例えば次のようになったとします(※実際の分割はモデルのトークナイザー依存で、必ずしもこの通りになるとは限らない)。

> 私 / は / タロウ / です / 。

を前提とし、以降の説明をします。

### Request Latency
リクエストを送ってから、最終応答(生成完了)を受け取るまでの総時間。

#### ☆ イメージ

`Request Latency = 最後のトークン「。」が出力された時刻 − リクエストを送った時刻`

### Time to First Token(TTFT)

リクエストを送ってから、最初の応答(1トークン目)を受け取るまでの時間。リクエスト1件につき1つの値が記録される。

#### ☆ イメージ

TTFTとは、

`TTFT = 「私」が出力された時刻 − リクエストを送った時刻`

となる。

#### 💡この指標のうまみ

- ユーザー体感の「待たされ感」を数値化できる
- ボトルネックが「入力処理」か「生成処理」かを切り分けられる
  - TTFTだけが遅い → 入力が長すぎる、またはprefillの処理効率が悪い(GPU・エンジンの問題)
  - TPOT(1トークンあたりの生成時間)だけが遅い → デコード(生成)処理自体が遅い、バッチサイズが大きすぎる等
- エンジン・インフラの比較が定量的にできる

####  追記:TTFTを計算する仕組み

そもそも、どうやってTTFTを計算しているのか気になったので調べた。

この指標を計算する際は、「ストリーミング(SSE: Server-Sent Events)」という仕組みを使っている。LLMがすべての出力を出し切ってから一括で返すのではなく、トークンごとなど細かい単位で逐次返してくれる。この仕組みのおかげでTTFTが計算できる。

なお、このストリーミング形式のリクエスト/レスポンスの形は、もともとOpenAI APIが採用したことで事実上の標準になったもので、今ではvLLMやTensorRT-LLMなど他のLLMサーバーも同じ形式(OpenAI互換API)に対応するようになっている。


[参考根拠サイト](https://leaysgur.github.io/posts/2023/04/12/171850/)


### Time to Second Token
ストリーミングで1つ目の応答を受け取ってから、2つ目の応答を受け取るまでの時間。TTFTと似ているが、「生成が本格的に始まった直後の速度」を見る指標。TTFTだけでは分からない、生成初速のブレを捉えられる。


#### ☆ イメージ
このときTTST(Time to Second Token、独自の略称)とは、

`TTST = 「は」が出力された時刻 − 「私」が出力された時刻`

#### 💡この指標のうまみ
* 「TTFTは速いのに、そのあとの出だしでカクつく」という現象があった場合、それを検知するための指標
* TTFTやITLほど頻繁に使われる指標ではなく、どちらかというとエンジン開発者・チューニング担当者向けの「デバッグ用の細かい指標」という位置づけ

### Inter Token Latency(ITL)　とTime Per Output Token（TPOT）
1つのリクエスト内で、連続する応答(トークン)同士の間隔時間を、生成されたトークン数で割った値。ストリーミング時の「文字が出てくる滑らかさ」を表す。ITLとTPOTは、ほぼ同義。

#### ☆ イメージ
ITLは厳密にトークン間の間隔を平均したもの。

`ITL = (最後のトークンが出力された時刻 − 最初のトークンが出力された時刻)  / (Total Output Tokens − 1)`

TPOTは、ITLよりざっくりした、トークンごとの平均時間を計算したもの。

`TPOT = (最後のトークンが出力された時刻 − 最初のトークンが出力された時刻)  / (Total Output Tokens)`

※ ITL・TPOTという名称・計算式は、ツールやドキュメントによって分母の取り方(トークン数か、トークン数−1か)が統一されていないことがあるため、複数ツールの数値を比較する際は注意が必要!

#### 💡この指標のうまみ
* ストリーミング中の「読みやすさ・滑らかさ」を数値化
* コスト計算の基礎データになる


### Request Throughput
ベンチマーク全体で完了したリクエスト数を、ベンチマーク全体の実行時間で割った値。「秒間何件のリクエストを捌けたか」を示す、こちらも全体で1値のみ。


#### ☆ イメージ

```
Request Throughput = 完了したリクエスト数 / 実行時間
                    = 50件 / 25秒
                    = 2.0 リクエスト/秒
```

#### 💡この指標のうまみ
* このサーバーに、実際どれくらいの人数が同時にアクセスしても大丈夫かを定量的に評価できる。
    * 実際どう使うか例(GPU増強の判断につながる流れ)
        * 想定ユーザー数を見積もる
            * 例:「ピーク時に1秒あたり10件のリクエストが来るサービスにしたい」
        * Request Throughputを計測する
            * genai-perfで計測したら「今のサーバー構成だと2.0リクエスト/秒までしか捌けない」とわかった
        * ギャップに気づく
            * 「10件/秒 来るのに、2件/秒しか捌けない」→ 明らかに処理能力が不足している
        * 対策を打つ
            * GPUを追加する(並列でPodを増やす、いわゆるスケールアウト)
            * より高性能なGPUに変える(スケールアップ)
            * バッチサイズやエンジン設定をチューニングする
            * モデルを量子化して1リクエストあたりの処理を軽くする
        * 再度計測して確認する
            * 対策後にもう一度Request Throughputを測り、「10リクエスト/秒に届いたか」を検証する


### Output Token Throughput
ベンチマーク全体で生成された出力トークンの総数を、ベンチマーク全体の実行時間で割った値。サーバー全体としての集約スループットであり、個々のリクエストではなくベンチマーク全体で1つの値しか出ない(平均・min/max等の分布は取らない)。

#### ☆ イメージ
```
Output Token Throughput = 総出力トークン数 / 実行時間
                        = (50件 × 100トークン) / 25秒
                        = 5000トークン / 25秒
                        = 200 トークン/秒
```
これを算出することでサーバーが生成した文章の総量を図ることができる。

#### 💡この指標のうまみ

* 「量子化やチューニングの効果」を定量的に確認できる
    * イメージとしては下記
        * チューニング前:Output Token Throughput = 400トークン/秒
        * チューニング後:Output Token Throughput = 600トークン/秒

* 「モデル・エンジン・GPUの純粋な生産能力」を横比較できる
    * Request Throughput:出力が短いほど件数が稼げて数字が良く見え、不公平な比較になりがち
    * Output Token Throughput:トークン量で測るため、出力長に左右されず公平に比較できる
        → エンジンやGPUの純粋な生成力を比べるならこちら


### Output Token Throughput Per User
1リクエストあたりの出力トークン数(最初の1トークン分を除く)を、生成フェーズにかかった総時間で割った値。「そのユーザーから見て、1秒あたり何トークン出力されたか」という体感速度に近い指標。

#### ☆ イメージ
```
Output Token Throughput Per User(太郎さんだけ) = (太郎さんの出力トークン数 − 1) / 太郎さんの生成時間
                                              = (100 − 1) / 2秒
                                              = 49.5 トークン/秒
```

* 整理すると
    * 50人が同時にサーバーを使っていても、Output Token Throughput Per Userは「その中の1人」に絞って計算します。
    * 　実際に画面の前でチャットを見ている人が体感する「文字が出てくる速さ」そのものです。

#### 💡この指標のうまみ
* ユーザー体感の速さを直接把握できる
* 混雑時の体感劣化を検知できる
* 同時接続数の上限を見積もれる
